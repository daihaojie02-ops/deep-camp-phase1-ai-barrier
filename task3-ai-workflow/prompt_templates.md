## 模板 1：任务拆解 Prompt

### 用途
将模糊的调研需求拆解为可执行的具体步骤和维度。

### 模板

```
你是一个技术调研方法论专家。你现在要帮助我设计一次技术调研方案。

【调研主题】：{调研主题，如"2026年AI Agent框架生态调研"}

【我的背景】：
- 技术水平：{新手/有经验/专家}
- 已知信息：{我目前知道的}
- 信息盲区：{我目前不清楚的}
- 个人约束：{如：偏好Python生态、Token预算有限、非.NET环境}

【调研目标】：{最终要做什么决定，如"选一个框架开始学习"}

【候选列表】：{已知候选，可以为"待发现"}

请按以下步骤完成调研方案设计：

1. **维度拆解**：列出本次调研需要覆盖的关键维度（至少 8 个）
   - 每个维度说明"为什么这个维度对我的决策重要"
   - 区分"硬数据维度"（可通过API/公开数据验证）和"软判断维度"（需人工判断）

2. **信息源规划**：针对每个维度，建议信息获取渠道
   - GitHub API 能覆盖哪些维度
   - Web搜索/Firecrawl 需要搜索什么关键词
   - 哪些维度只能通过实际运行/社区讨论/个人判断来获取

3. **风险预警**：根据我的盲区，预判哪些结论可能不可靠
   - 每个风险点标注"如果信息不足，建议如何处理"

4. **输出结构建议**：建议最终报告的分章结构

注意：
- 如果某个维度的信息获取方式不明确，请标注"需人工确认"并给出建议
- 不要编造具体数据，只设计框架
- 如果我的约束条件与调研目标存在冲突（如：预算有限但候选需要付费API），请明确指出来
```

### 输入/输出

| 输入 | 输出 |
|------|------|
| 调研主题 + 个人背景 + 约束条件 + 候选列表 | 调研方案：维度列表 + 信息源规划 + 风险预警 + 报告结构建议 |

### 使用注意

- **需要人工复核**：是。AI 建议的维度可能遗漏领域特有的评估标准
- **常见调整**：我在 Task 002 实际使用时，AI 建议了 12 个维度，我去掉了 2 个不适用于"新手视角"的维度（contributor 深度分析、架构对比），增加了"学习资源质量"
- **关键原则**：维度不是越多越好，每个维度必须能回答"这个维度如何影响我的决策"

### 设计逻辑

这个模板的设计核心是**约束前置**。普通 AI 使用方式是"帮我调研一下 XX 框架"，AI 会输出一个百科式对比。这个模板通过强制前置填写"个人背景、已知信息、盲区、约束条件"，让 AI 从"通用调研"变成"个性化调研"。Task 002 的经验告诉我：缺少约束条件的调研报告就是信息堆砌，有约束条件的调研才有决策价值。

## 模板 2：内容生成/处理 Prompt

### 用途
对原始信息进行结构化分析和深度处理，产出对比分析内容。

### 模板

```
你是一个技术分析专家。我将给你一份关于 {调研主题} 的原始信息集合。

【原始信息】
{粘贴 Firecrawl/GitHub API 抓取的原始内容}

【分析要求】
请对以下候选进行深度分析：

候选列表：{候选1, 候选2, 候选3, ...}

对每个候选，按以下结构分析：

## {候选名称}

### 1. 定位与核心概念（1-2句话）
- 这个框架解决什么核心问题？它把自己定位为什么角色？

### 2. 关键数据（从原始信息中提取，标注数据来源和获取日期）
- GitHub Stars / Issues / PR / Last Commit
- 主要语言 / 许可证
- 最近 3 个月的关键更新

### 3. 核心特点（3-5条，每条一句话）
- 什么特性让它和别的框架不一样？

### 4. 社区健康度评估
- Issue 响应速度（从数据推断）
- 文档质量（从内容判断）
- 是否有明确的维护团队

### 5. 学习曲线判断
- 最小可运行示例的代码量
- 概念学习成本（需要预先理解多少前置知识）
- 官方教程/入门资源的可获取性

### 6. 适用场景
- 最擅长解决什么问题
- 不适合解决什么问题

### 7. 局限与风险
- 已知的架构限制
- 社区反馈的常见痛点
- 版本稳定性

【重要约束】：
- 所有数据必须标注"来源：xxx，获取日期：xxx"
- 如果原始信息中缺少某个维度的数据，标注"信息不足，无法判断"
- 不要编造数据或推论
- 区分"事实陈述"和"分析判断"：事实用"根据数据显示"，判断用"基于以上数据我认为"
```

### 输入/输出

| 输入 | 输出 |
|------|------|
| 原始信息集合 + 候选列表 | 每个候选的 7 维度结构化分析 |

### 使用注意

- **需要人工复核**：是。必须逐项核对数据来源和提取日期
- **Task 002 实际坑**：AI 有时会把"社区讨论中的主观评价"当作"客观数据"来引用。需要在原始信息中区分"第一手数据"（API抓取）和"第二手观点"（博客文章）
- **常见调整**：如果候选多于 4 个，建议分批处理避免 AI 输出质量下降

### 设计逻辑

这个模板的关键创新是**强制标注来源和获取日期**。Task 002 的经验：AI 生成的分析看起来很有道理，但如果没有来源标注，3 天后我自己都不知道哪些数据是可验证的、哪些是 AI 可能编造的。来源标注不是为了格式好看，是为了**可追溯性**——这是信息检索场景的核心质量要求。

## 模板 3：质量检查 Prompt

### 用途
对 AI 产出的调研报告进行系统性质量检查，发现编造、遗漏、逻辑漏洞。

### 模板

```
你是一个严格的技术审稿人。请逐项检查以下调研报告的质量。

【待检查报告】
{粘贴完整报告内容}

【原始输入材料】
{附上用于生成报告的原始信息，供交叉验证}

请按以下 10 个检查项逐一检查，每项给出"通过/不通过 + 具体证据"：

| # | 检查项 | 通过？ | 证据/问题 | 严重度 |
|---|--------|--------|----------|--------|
| 1 | 所有数据是否标注了来源？ | | | |
| 2 | 是否存在原始信息中找不到的数据（疑似AI编造）？ | | | |
| 3 | 每个候选的"事实陈述"和"分析判断"是否区分明确？ | | | |
| 4 | 个人约束条件是否在推荐理由中体现？ | | | |
| 5 | 是否存在模板化表达（如"具有强大的生态"、"拥有活跃的社区"无具体数据支撑）？ | | | |
| 6 | 信息盲区是否标注清晰（哪些无法判断、为什么、影响多大）？ | | | |
| 7 | 报告结论是否与数据一致（有无数据说A但结论说B的矛盾）？ | | | |
| 8 | 所有候选是否在同一尺度下对比（有无对某候选用"社区活跃"对另一候选用"stars数量"）？ | | | |
| 9 | 是否包含可执行的下一步建议（不只是"推荐XX"，而是"从XX开始，第一步做YY，预计时间ZZ"）？ | | | |
| 10 | 如果我是目标用户，读完后能否做出决策？ | | | |

【输出要求】：
- 每项不通过必须给出"具体哪里有问题 + 建议如何修改"
- 最后给出总体通过/不通过判断
- 如果不通过，列出需要修改的具体段落和修改建议
- 如果发现事实性错误，标出原文 + 正确信息（如果正确信息在原始材料中能找到）
```

### 输入/输出

| 输入 | 输出 |
|------|------|
| 完整报告 + 原始输入材料 | 10 项逐条检查结果 + 修改建议列表 |

### 使用注意

- **需要人工复核**：是。AI 检查 AI 有盲区——两个 AI 可能犯同样的错误（如都对某个框架的定位有误解）。第 2 项"疑似编造"的检查结果需要人工二次确认
- **关键原则**：这个 Prompt 的核心价值不是让 AI 代替人检查，而是让 AI 做**第一轮粗筛**——把明显的问题（数据缺失、来源缺失、矛盾）先揪出来，人工再做第二轮精审
- **Task 002 的实际效果**：质量检查 Prompt 在 Task 002 中发现了 3 个问题——1 处来源未标注、1 处模板化表达、1 处结论与数据的轻微矛盾。如果没有这个环节，这些问题可能被漏掉

### 设计逻辑

这个模板的精髓在**第 8 项**："所有候选是否在同一尺度下对比"。这是 Task 002 踩过的坑——AI 倾向于对不同的候选用不同的描述语言，比如对 A 框架说"社区活跃"、对 B 框架说"stars 数量多"，读者无法对比。统一尺度是信息检索质量的最低门槛，但恰恰是最容易被 AI 绕过的。

## 三个 Prompt 的协作关系

```
模板 1（任务拆解）
  ↓ 产出：调研方案 + 维度列表 + 信息源规划
  ↓
信息检索（Firecrawl + GitHub API）—— 按模板1的信息源规划执行
  ↓ 产出：原始信息集合
  ↓
模板 2（内容生成/处理）
  ↓ 产出：结构化分析草稿
  ↓
人工判断注入（Step 3）—— 注入约束条件 + 盲区标注
  ↓ 产出：含判断的分析稿
  ↓
模板 2（再次使用，生成对比矩阵和完整报告）
  ↓ 产出：完整报告初稿
  ↓
模板 3（质量检查）
  ↓ 产出：问题列表 + 修改建议
  ↓
人工精审 + 修改
  ↓ 产出：最终报告
```
