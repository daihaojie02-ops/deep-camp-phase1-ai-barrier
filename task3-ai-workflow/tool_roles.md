## 工具分工总表

| 工具名称 | 负责环节 | 输入内容 | 输出结果 | 为什么适合 | 风险或限制 |
|---------|---------|---------|---------|-----------|-----------|
| **Firecrawl MCP** | Step 1: 信息检索 | 搜索关键词 + URL列表 | 结构化网页内容（标题、正文、来源URL） | 能同时搜索 web/articles/github 多源，返回结构化 Markdown 而非零散链接，大幅减少手动整理时间 | 搜索深度依赖关键词设计；可能遗漏非英语源；部分站点有反爬限制 |
| **Claude Code (Claude)** | Step 2: AI 结构化提取 + Step 4: 对比矩阵生成 + Step 6: 报告输出 | 原始信息 + 分析维度要求 + 输出模板 | 维度拆解草稿、对比矩阵、结构化报告初稿 | 长文本处理能力强（能一次性消化多个框架的文档），结构化输出一致性好；支持 MCP 工具链，可以在分析过程中实时抓取补充数据 | 可能产生看似合理但数据偏差的内容；需要人工核查关键数字和引用 |
| **GitHub CLI (`gh`)** | Step 1: 辅助信息检索 | 仓库名（如 crewAIInc/crewAI） | stars/issues/PR/last_commit/contributors 等结构化数据 | 直接调用 api.github.com（国内不受 GFW 影响），返回 JSON 可程序化处理；数据可重复抓取验证 | 受 GitHub API rate limit 限制（未认证 60次/h）；只覆盖 GitHub 生态的数据维度 |
| **Playwright MCP** | 辅助截图 + 页面验证 | 网页 URL | 页面截图（用于提交材料证明） | 能直接截 GitHub 页面/搜索结果作为原始材料证明，比文字描述更有说服力 | 浏览器启动有开销；部分页面加载慢需要等待；截图尺寸需手动调整 |

## 工具选择逻辑：为什么不加更多？

Task 002 的实际经验告诉我：**工具越多，切换成本越高，越容易在工具之间丢失信息上下文。**

- Task 002 用了 5 个工具（Firecrawl + Claude + GitHub API + Playwright + WebSearch），但核心产出 80% 来自 Firecrawl + Claude 两个工具
- WebSearch 搜索结果与 Firecrawl 高度重叠，属于冗余
- 因此 Task 003 工作流精简为 **4 工具核心集**

选择标准不是"这个工具能做这个吗"，而是：

| 过滤条件 | 说明 |
|---------|------|
| **不可替代性** | 缺了这个工具，某个环节就无法完成或质量大幅下降 |
| **信息不重叠** | 和已有工具抓取的信息源不高度重复 |
| **切换成本可接受** | 从我当前工作环境切到这个工具的成本（登录/配置/学习）不高 |
| **输出可直接用于下一步** | 工具的输出格式能直接喂给下一个环节，不需要中间手动转换 |

**为什么没选以下工具：**

| 工具 | 不选的原因 |
|------|-----------|
| Google/Bing 搜索 | GFW 限制 + Firecrawl 已覆盖 web 搜索 + 结果格式不如 Firecrawl 结构化 |
| NotebookLM | 适合长文档深度阅读，但本工作流场景是"多源碎片信息整合"，不是"单长文档分析" |
| Perplexity | 搜索结果不可复现（每次相同问题可能不同答案），违反工作流"可复现"原则 |
| ChatGPT/Gemini 独立使用 | Claude Code 已集成分析能力，多工具切换增加信息丢失风险；但 Gemini 在特定多模态场景可作补充 |
