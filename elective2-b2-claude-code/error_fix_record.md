## 修复记录 1：数据从示例改为真实数据

**出现的问题：** Claude Code 生成的初始 HTML 使用了占位示例数据（"Task 1: ..."、"必修任务 1"等），而不是我真实的 DEEP 营任务数据。

**我如何发现的：** 浏览器打开 index.html 后立即发现——任务名称、评分、截止日期全都不对。

**我向 Claude Code 如何描述：**
```
把表格里的示例数据替换成我的真实数据：
- 001 AI 壁垒方向卡，状态已完成，评分 A+，截止 7/18
- 002 AI 真实场景应用实验，状态已完成，评分 A+，截止 7/18
- 003 AI 工作流与 Agent 雏形，状态已完成，评分 A，截止 7/19
- X1 AI 加速器，状态已完成，评分 A，截止 7/18
- B1 AI 信息壁垒报告，状态进行中，截止 7/18 18:34
- B2 Claude Code 真实问题解决，状态进行中，截止 7/19 10:50
- B3 AI 辅助 Python 零基础入门，状态进行中，截止 7/19 18:11
```

**Claude Code 给出的修改：** 更新了表格中的任务名称、状态、评分和截止日期。

**验证成功：** 刷新浏览器后所有数据显示正确。

**学到了什么：** Claude Code 生成的内容总是带着"默认数据"——示例名称、假日期。**在 Prompt 中提前提供真实数据可以让第一版更接近目标，减少迭代次数。** 这个教训适用于所有 Claude Code 使用场景。

## 修复记录 2：优化进度条计算逻辑

**原始结果的问题：** 进度条写死了 `width: 57%` 和 `4/7 已完成`。如果后续有新任务完成，需要手动修改两处——容易不一致。

**我不满意的地方：** 数据只有 7 个任务，但硬编码的进度显示不优雅——应该一眼看出逻辑。

**我让 Claude Code 如何修改：**
```
进度条比例：4 个必做已完成 + 0 个选做已完成 = 4/7 ≈ 57%。
写成百分比时确保和分数一致。
```

**修改后效果：** 进度条宽度和文字保持一致，4/7 ≈ 57%。

**从这次"优化"中学到的：** 如果任务数量变化频繁，应该用 JS 动态计算完成率。但当前只有 7 个任务且固定，硬编码是可接受的——**不要过度工程化一个展示页面。**

## 修复记录 3：颜色语义检查

**原始结果的问题：** 截止日期全部用同一种颜色——快到期和还有时间的没有区分。

**我不满意的地方：** B1 截止 7/18（只剩 1 天）、B2 截止 7/19（只剩 2 天），应该用红色警告。B3 截止 7/19 18:11（稍宽松），可以保持默认色。

**修改后：** B1 和 B2 的截止日期加了 `.deadline-soon` class（红色），B3 保持默认。

**学到了什么：** UI 设计中的"信息层级"很重要——相同颜色意味着相同重要度，不同颜色创造对比和注意力引导。即使是只有我一个用户的页面，这种设计细节也让信息获取更快。
