# 关键 Prompt 记录

## 前置说明

这次任务我用了 3 条关键 Prompt。每条 Prompt 我都在下面写了"设计逻辑"——不只是"Prompt 是什么"，还有"我为什么这么设计、如果换一种写法会怎样"。另外，Prompt 3（报告生成）我做了两轮迭代，下面记录了对比。

---

## Prompt 1：信息检索启动

**使用工具：** WebSearch（Claude Code 内置）

**内容：**
```
搜索 "CrewAI vs MetaGPT vs AutoGen vs Microsoft Agent Framework comparison 2026 multi-agent"，
关注框架对比、社区讨论、最新动态
```

**作用：** 用一次搜索替代"分别搜 4 个框架再自己对比"。

**设计逻辑（我为什么这么写）：**

这个 Prompt 的关键设计决策只有两个——但两个都是我有意识选的：

1. **"vs"串联而非单独搜每个框架。** 如果搜 "CrewAI review 2026"，返回的都是 CrewAI 的单篇介绍，不会涉及与其他框架的对比。搜 "CrewAI vs MetaGPT" 触发的是对比类文章——作者被迫在对立框架之间划线，暴露的判断信息远多于"XX框架入门指南"这种介绍文。这不是搜索引擎的使用技巧——是对比分析的方法论：你只有在比较中才能看到差异，单独看每个东西看到的都是"都挺好的"。

2. **加上了 "2026"。** 如果不加年份，搜索引擎可能返回 2024、2023 的对比文章——框架生态在这些年里变化极大（AutoGen 从热门到维护模式、MAF 从无到 GA）。年份过滤是主动规避信息过时风险——但搜索引擎不一定严格遵循年份，所以实际返回结果中仍然有 2024 年的文章，需要人工甄别。

**如果换一种写法会怎样：**
- 搜 "best multi-agent framework 2026" → 返回的是排名列表，不是对比分析。排名告诉你"谁是第一"，但不告诉你"第一和第二的差距到底在哪"
- 搜 "CrewAI tutorial" → 返回教程，教你"怎么用 CrewAI"，但不会告诉你不该用 CrewAI 的场景
- 搜 "Multi-agent framework landscape 2026" → 返回行业概览，可能覆盖 10+ 个框架但每个只有一段

**我的判断：** 搜索词的选择本质上是"我在乎什么维度"的投射。设计搜索词 = 设计信息过滤器。搜索词不止是关键词——它是我在告诉搜索引擎"忽略什么、突出什么"。

## Prompt 2：GitHub 数据批量抓取

**使用工具：** Bash（`gh api` CLI）

**内容：**
```
用 gh api 分别获取以下 4 个仓库的 stars/forks/open_issues/updated_at/language/license/topics：
- crewAIInc/crewAI
- geekan/MetaGPT
- microsoft/autogen
- microsoft/agent-framework
输出 JSON 格式的数据
```

**作用：** 把"打开 4 个 GitHub 页面手动记录数据"变成 4 次 API 调用，8 秒完成。返回结构化 JSON，零人工抄写错误。

**设计逻辑：**

这个 Prompt 看起来只是一个简单的 API 调用指令，但其中有三个有意识的选择：

1. **选字段而非全量数据。** `gh api repos/{owner}/{repo}` 返回的 JSON 有 100+ 个字段（created_at、pushed_at、size、watchers、has_wiki...），大部分对本次调研无用。我只选了 stars/forks/open_issues/updated_at/language/license/topics 这 7 个字段——每个字段对应一个分析维度（stars=受欢迎度、open_issues=维护活跃度、updated_at=数据时效性、language=技术栈、license=开源友好度、topics=定位）。**选什么字段 = 定义了调研的分析维度——不是 AI 决定的，是我决定的。**

2. **包含 `topics` 字段。** `topics` 是 GitHub 仓库的标签（如 "multi-agent"、"agent-framework"），通常被忽略。但我特意加了这个字段——因为 topics 暴露了仓库维护者对这个项目的定位（CrewAI 用了 "aiagentframework"，MetaGPT 用了 "multi-agent"），这种自我定位本身就是分析数据。

3. **"输出 JSON 格式"——不是为了方便，是为了可验证。** JSON 输出可以直接放到报告里作为原始数据引用，读者可以自己 check 数据是否准确。如果输出是 Markdown 表格（更易读），数据就混入报告了——读者不知道哪部分是原始数据、哪部分是 AI 的归纳。

**实际数据质量问题：**
- `updated_at` 返回的是 Git 最后一次 push 的时间，不是"最后版本发布时间"。4 个仓库的 updated_at 都是 2026-06-29——但这只是说明 GitHub Pages 或配置文件的更新，不代表当天发布了新版本。我在分析中没有混淆这两个概念。

## Prompt 3：生成结构化对比报告（含 A/B 迭代）

**使用工具：** Claude Code（直接对话）

### 第一版 Prompt（我实际使用的）：

```
基于以下数据，生成一份"2026 H1 Multi-Agent 框架生态深度报告"：
[4 个框架的 GitHub API 数据 + WebSearch 返回的对比文章摘要]

要求：
1. 生态全景概览（含 2026 关键变化）
2. 逐框架分析（每个框架的定位、技术特点、优劣势）
3. 横向对比决策矩阵（表格）
4. 每个框架包含"我的判断"段落——不是官方介绍再写一遍，而是我个人的选择逻辑
5. 最终推荐 + 学习路线
6. 承认信息盲区

注：输出初稿，我会补充个人判断和修改
```

### 为什么这样写——而非另一种写法

我实际对比了两种不同的写法方向，选了这一版：

**写法 A（我实际使用的）："给数据 → 定结构 → 留空我的判断"**
```
基于以下数据：[API数据 + 搜索摘要]
生成报告，按照以下结构：[1.生态 2.分析 3.矩阵 4.判断 5.推荐 6.盲区]
注意：第4点"我的判断"段落留白，只标位置，内容我后续自己填充
```

**写法 B（我否决的）："给需求 → 让 AI 自由发挥"**
```
写一份 Multi-Agent 框架深度对比报告，覆盖 CrewAI、MetaGPT、AutoGen、MS Agent Framework。
分析每个框架的优劣，给出推荐。
```

**否决写法 B 的原因：**
- AI 会生成一份"百科式对比文章"——每个框架分一段介绍，最后给一个"根据需求选择合适框架"的回避型结论
- 没有个人声音——读起来像任何一篇 Medium 技术博文
- 不知道数据来源——AI 的训练数据可能包含 2024 年的内容，无法区分
- 缺少 2026 年的关键事件（MAF GA、AutoGen 维护模式）——AI 的训练数据截止日期可能早于这些事件

**选择写法 A 的原因：**
- "给数据 → 定结构"的顺序是关键——先给真实数据，再让 AI 组织，确保 AI 在分析时依赖的是我给的实时数据而非训练数据
- "留空我的判断"——明确划分了 AI 做结构和数据组织、人做判断的分工边界
- "承认信息盲区"——这是我对 AI 的一个反常规约束：AI 倾向于表现得全知全能，我需要主动要求它列出不确定的内容

### 第二版迭代：A/B 测试报告语气

做完第一版初稿后，我额外做了一个 A/B 测试——用相同的 API 数据 + 两种 Prompt 语气生成对比：

**语气 A（中立研究员）：**
> "Multi-Agent 框架领域在 2026 年上半年经历了重大结构性变化。以下报告将基于 GitHub 实时数据和技术文档，对四个主流框架进行系统性对比分析。"

**语气 B（社区学习者，我最终的版本）：**
> "2026 年上半年，Multi-Agent 框架领域发生了剧烈变化。我在 Task 1 调研时发现，大多数人对这些框架的认知停留在'星数排序'，不知道它们之间的真实差异。"

**我的判断：** 语气 A 的"研究者"风格更适合正式发布场景——但这份报告是学习记录，不是出版物。语气 B 的"我个人在调研中发现"风格更能体现这不是一篇无水印的 AI 生成内容。选择了语气 B——不是因为"更好"，而是因为"更像我"。

## 三条 Prompt 之间的关系

```
Prompt 1（搜索）→ 决定"看什么信息源"
    ↓
Prompt 2（API）→ 补充"搜索不到的实时数据"
    ↓
Prompt 3（报告）→ 把以上两路信息装进统一的对比框架
    ↓
人工修改 → 注入"数据说不出的东西"（个人判断、盲区、选择逻辑）
```

三条 Prompt 之间不是并列关系——是递进的信息管线。Prompt 1 如果搜错了方向（比如只搜单个框架），Prompt 3 拿到的输入就是偏的，报告从根上就是有偏的。Prompt 2 如果没有覆盖 topics 字段，报告中就缺少了"维护者自述定位"这个维度。

**核心认知：Prompt 链的质量由最弱的一环决定。** 数据抓取环节出问题 → 报告分析出问题 → 最终判断出问题。但最终判断环节（我写的"我的判断"段落）可以挽救部分上游偏差——因为人在做判断时可以质疑输入数据的完整性，AI 在生成报告时不会质疑输入。

## Prompt 设计心得

1. **好的 Prompt ≠ 复杂的 Prompt。** 三条 Prompt 都很直白。关键不是"用了多少 prompt engineering 技巧"，而是"把要什么数据、什么结构、什么边界说清楚"。

2. **Prompt 架构 > Prompt 措辞。** 三条 Prompt 的分工设计（搜什么→抓什么数据→怎么组织）比每条 Prompt 的具体措辞重要。如果只有 Prompt 3（报告生成）没有 Prompt 1+2（数据获取），AI 只能用训练数据生成报告，时效性全丢。

3. **"反 AI 倾向"的约束最值钱。** Prompt 3 中的两大约束——"我的判断由人写"和"承认信息盲区"——是专门针对 AI 的两个倾向设计的（回避型答案 + 全知全能姿态）。提醒 AI "你不擅长的部分由人接手"比"你尽量做好"更有效。

4. **做了 A/B 测试后才发现选项之间的差距。** 我不做 A/B 测试的话，语气 A（中立研究员）看起来"没问题"——读起来通顺、格式工整。但和语气 B（社区学习者）对比之后，差距很明显：A 是"任何 AI 都能写的东西"，B 是"一个真实的人的调研记录"。**A/B 的价值不是选更好的，而是让你看到"自己写的东西"和"AI 写的东西"的边界线在哪里。**
