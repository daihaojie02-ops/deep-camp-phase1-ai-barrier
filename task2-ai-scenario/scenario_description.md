# 真实场景说明

## 场景类型
**信息检索场景** — 深度技术调研报告

## 1. 这个场景具体是什么？

2026 年上半年，Multi-Agent 框架领域发生了剧烈变化。四个主要框架的格局完全重组：

- **AutoGen（59K⭐）→ 2025 Q4 进入维护模式。** 两年前最受关注的 Multi-Agent 框架，现在停止开发，微软官方推荐迁移到新框架。
- **Microsoft Agent Framework（11.7K⭐）→ 2026.04 GA。** AutoGen + Semantic Kernel 合并为统一的生产级平台，继承了两个旗舰项目的积累。
- **CrewAI（54.5K⭐）→ 2025 年增长 170%。** 从 20K⭐ 增长到 54.5K⭐，成为增长最快的独立 Agent 框架。
- **MetaGPT（69.1K⭐）→ ICLR 2025 Oral。** 学术路线持续吸引关注，但实际落地场景受限。

**我要做的事：** 不靠几篇过时的 Medium 文章拍脑袋，而是用 GitHub API 实时数据 + 多源对比文章做一次系统性的社区生态调研——搞清楚"哪些框架在增长、哪些在衰退、各自适合什么场景、我作为新手应该从哪开始"。

## 2. 这个问题真实存在在哪里？

**在我自己的学习决策中。** 我在 Task 1 自评 Multi-Agent 编排 1/5——这个 1/5 不只是"经验不足"，而是"连从哪开始都不知道"。

具体来说：如果我现在需要学 Multi-Agent，面对 4 个选择——CrewAI、MetaGPT、AutoGen、MAF——我应该选哪个？不靠系统调研，我的决策流程会是：
- 打开 Google → 搜 "best multi-agent framework 2026" → 看第一篇对比文章
- 如果那篇文章是 2024 年写的 → 它不会告诉我 AutoGen 已停止维护
- 如果那篇文章是 CrewAI 用户写的 → 它会有选择偏差
- 30 分钟后我拍脑袋选一个 → 可能一周后发现选错了

**这个问题不是特殊场景——任何在 2026 年想了解 Multi-Agent 框架的人都面临同样的信息混乱。** 框架选择本身只是冰山一角——真正的问题是"在信息源不确定的情况下如何做出有依据的判断"。

## 3. 是我自己的问题，还是别人也可能遇到？

**两者都是。**

**是我自己的问题：** Task 1 暴露了我对 Multi-Agent 框架的真实认知水平——知道名字和星数，但说不出差异。这个信息差来自我没有做过系统调研，只停留在"看到就记住"的被动学习层面。

**别人也可能遇到：** 对任何在 2026 年搜索 "best multi-agent framework" 的人来说——
- Google 搜索结果中排名靠前的文章大概率是 2024-2025 年的，不包含 MAF GA 和 AutoGen 维护模式两个关键事件
- GitHub 星数排序（MetaGPT > AutoGen > CrewAI > MAF）会误导新手——星数反映的是历史关注度，不是当前状态
- 社区讨论分散在 Reddit、Twitter、Hacker News、技术博客、知乎——没有系统整理

**这个问题的一个典型受害者画像：** 一个开发者在 2026 年 6 月搜到一篇 2024 年 12 月的文章推荐 AutoGen，然后在 AutoGen 上花了两周学习，最后发现官方已经放弃了这个项目。这不是虚构——这就是信息差不纠正的后果。

## 4. 不用 AI 时，我原本会怎么解决？

**典型流程（预估 30 分钟）：**

1. 打开 Google → 搜 "CrewAI vs MetaGPT" → 看第一个搜索结果（大概率是 Medium 上的对比文章）→ 约 15 分钟
2. 打开 GitHub 依次看 4 个仓库 → 手动记录 stars/forks → 约 10 分钟
3. 凭文章结论 + 星数高低做决定 → 约 5 分钟

**这个流程的实际产出：** 一个脑内印象（"CrewAI 好像不错，星数第三"）——没有结构化记录，没有交叉验证，没有可追溯的判断逻辑。一周后如果有人问我"你为什么不选 MAF？"，我只能回"网上说 CrewAI 适合新手"——无法给出基于自身情况的具体判断。

**更关键的：** 30 分钟的"快速决策"看起来省了时间——但实际上如果选错了框架，后续浪费的时间远超 30 分钟。一份系统调研投入 97 分钟，回报是"选对框架后节省的数百小时学习时间"。

## 5. 不用 AI 时，困难在哪里？

**困难 1：信息源质量不可控。** 搜索引擎的排序 ≠ 信息质量。一篇 SEO 优化得很好的文章可能在每个框架分析段末尾加一个通用推荐——而一篇真正有判断力的对比文章可能因为域名权重低排在第三页。

**困难 2：无法批量获取实时数据。** 手动打开 4 个 GitHub 页面看数据不是不行——但很慢，而且容易只记录 stars 而忽略 open_issues（AutoGen 的 922 个积压 issues 是关键信号）。更严重的是——手动浏览看到的是"当前快照"，看不出历史变化趋势。

**困难 3：没有交叉验证机制。** 看一篇对比文章 + 4 个 GitHub 页面 = 5 个信息源，但没有交叉验证。文章说 CrewAI 适合新手——但我无法区分"作者的观点"和"客观事实"。当多个来源对一个框架的评价一致时，可信度才上升；当 AutoGen 被提及的频率下降 + open_issues 上升 + 维护模式公告——三个信号指向同一个结论时，才敢确认。

**困难 4：结构化的认知负担。** 看 4 篇对比文章后，脑中有一堆碎片信息——CrewAI 角色驱动、MetaGPT SOP 驱动、AutoGen 对话驱动、MAF 图编排——但很难把它们放到同一个对比框架下。"列维度再填数据"这个思维过程，不用 AI 的情况下的确可以手动做——但更可能在"太麻烦了"的临界点上放弃。

## 6. 这个场景为什么适合作为 AI 应用实验？

**原因 1：信息检索是 AI 的强项——但需要人有判断地使用。** Firecrawl/WebSearch 能快速扫描多个信息源，但不是"搜到什么就信什么"——人需要筛选哪些文章可信、哪些数据需要交叉验证。**AI 做广度，人做深度——这是信息检索场景的最佳分工。**

**原因 2：结构化输出体现 AI 的工具价值。** "把 4 个框架的信息装进同一个对比框架"——这需要一次处理大量信息、保持格式均匀、不遗漏维度。一个人手动做这件事会犯三个错误：格式不一致（第一个框架写得很详细，写到第四个时偷懒）、遗漏维度（想不全 10 个对比维度）、结构僵硬（写到一半发现结构要改但来不及）。AI 在这些问题上没有人的疲劳感。

**原因 3：有"不用 AI vs 用 AI"的明确对比空间。** 这个场景不用 AI 也能做（30 分钟快速查阅），用 AI 后也能做（97 分钟系统调研）——两者的产出差异巨大，但都能产出结果。这种"两个版本同时存在、可以被对比"的场景，比"AI 能做、人做不了"的场景更有实验价值——因为它直接展示了 AI 的增值点在哪、人的增值点在哪。

**原因 4：产出可公开验证——不是自说自话。** 报告的结论（我选 CrewAI）可以被任何人质疑——只需检查 GitHub API 数据是否准确、对比矩阵的逻辑有无漏洞、个人判断是否基于真实的个人约束。这种"可被挑战的判断"远比"AI 生成的通用结论"有价值。
