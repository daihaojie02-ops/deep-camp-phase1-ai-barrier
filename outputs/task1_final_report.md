# AI 壁垒方向卡 — 戴昊杰

## 1. 我为什么要学习 AI？

我学习 AI 不是出于"大家都在学"的焦虑，而是基于一个清晰的判断：**软件工程的生产力边界正在被 AI 重写**。

过去半年，我从手动编写代码逐步转向用 Claude Code 等 AI Agent 完成项目搭建、自动化脚本、网页开发和数据分析。一个原本需要 3-5 天完成的 MCP Server 集成项目，现在可以在半天内完成设计、实现、调试和部署。这种效率跃迁不是线性优化，而是范式转换。

我的核心动机：
- **工程效率最大化**：AI Agent 不是代码补全工具，是一个可以独立执行任务、调用工具链、完成闭环交付的生产力操作系统；
- **不可替代性构建**：当 GPT、Claude、Gemini 人人可用时，差距不在"会不会打开 ChatGPT"，而在"能不能把 AI 编排成自己的工程系统"；
- **真实能力积累**：DEEP 营的任务体系要求可检查、可复现、可交付，这符合我对"真能力"的理解。

## 2. AI 未来可能在哪些方面改变学习、工作、比赛、项目或创业？

| 领域 | 当前变化 | 未来趋势 |
|------|----------|----------|
| 学习 | AI 辅助信息检索、知识解构、学习路径规划 | 个性化 AI 导师系统，基于个人知识图谱的精准补缺 |
| 工作 | AI Agent 替代重复性操作，从"人做流程"变为"人定流程" | Agent 间协作编排，人类角色从执行者变为架构师和审核者 |
| 比赛 | AI 黑客松、Agent 算法大赛成为主流赛道 | 比赛考察点从"算法精度"转向"Agent 编排能力" |
| 项目 | Claude Code 驱动的全栈闭环：方案→实现→Eval→部署 | 单人工程能力被 AI 放大 10 倍，小团队能承担大项目 |
| 创业 | 最小交付成本大幅降低：一个 Agent 就能完成 MVP | "会用 AI 的 1 个人"成为最小作战单元 |

## 3. 我现在和 AI 前沿使用者之间最大的差距是什么？

审视自己的现状，差距主要在三个维度：

**工具链深度**：我能熟练使用 Claude Code 和基础的 MCP Server，但对比前沿使用者——他们把 Claude Code 配置成了完整的开发操作系统，有自定义 Skills、自定义 Hooks、多 Agent 编排流水线。我的工具使用还停留在"对话式编程"层面，没有真正工程化。

**信息获取速度**：前沿使用者有稳定的 AI 信息雷达——GitHub Trending、Hugging Face Papers、Anthropic 发布渠道、Twitter/X 的 AI 信息流。我的信息获取更多是被动接收，缺少主动信息猎取和判断机制。

**领域耦合深度**：纯 AI 工具使用者壁垒很低。但"AI + 某个垂直领域"的耦合壁垒很高。我现在还没有把 AI 能力和我准备深耕的领域（工程自动化、Agent 系统架构）做深度融合。

## 4. 我目前最需要解决的 3 个问题

1. **工程化程度不足**：Claude Code 的 CLAUDE.md、Skills、Hooks、MCP Server 编排还不够系统化，没有形成可复用、可分享的工程规范；
2. **输出不可见**：做了不少项目但没有形成公开作品集（GitHub、技术文章、Demo 页面），外部不可验证；
3. **信息雷达缺位**：缺少定期追踪 AI 前沿信息的机制，容易陷入"用熟悉的工具反复做熟悉的事"的舒适区。

## 5. 我希望 AI 优先辅助我完成哪些具体任务？

- **代码项目全流程闭环**：从需求分析 → 架构设计 → 代码实现 → 测试验证 → 部署交付，全部由 Claude Code 作为主引擎驱动；
- **MCP Server 开发与编排**：针对特定场景（数据分析、网页监控、内容处理）快速搭建专用 MCP Server；
- **自动化工作流搭建**：用 Agent 替代日常手动重复操作（如数据采集、报告生成、文件处理）；
- **技术内容产出**：博客文章、项目文档、技术复盘的结构化生成。

## 6. 我未来想建立的个人 AI 壁垒是什么？

我的长期 AI 壁垒方向是：**AI Agent 系统工程化能力**。

具体来说：
- 不是"会用一个 AI 工具"，而是"能把多个 AI Agent 编排成可运行的工程系统"；
- 不是"写 Prompt 让 AI 做事"，而是"设计 Agent 间的协作协议、评估标准和优化闭环"；
- 不是"AI 辅助写代码"，而是"以 AI 为主引擎、以我为主架构师，完成从 0 到 1 的项目交付"。

这个方向上，门槛会越来越高：Claude Code 的 Skills 系统、MCP 协议、Agent 编排框架都在快速演进。先发优势和持续积累的复利效应在这个领域非常明显。

## 7. 如果别人也会使用 ChatGPT、Claude、Gemini、Codex，我准备如何和他们拉开差距？

1. **工程深度**：不只"用 AI 聊天"，而是把 AI 编排成自己的操作系统级工具。CLAUDE.md 不是一份说明文件，是一个持续进化的工程规范；
2. **可验证产出**：所有的学习成果都以 GitHub 仓库、可运行项目、技术文章的形式公开存在，而不是"我觉得我学会了"；
3. **领域耦合**：把 AI 能力和具体的领域需求（Web 自动化、数据分析、内容生产）做深度绑定，形成"AI + 领域"的双重壁垒；
4. **持续复利**：每做一个项目就沉淀为 Skill、MCP Server 或工程模板，知识资产持续积累，不自废武功。

## 8. 这个暑期结束时，我希望自己在哪些方面发生明显变化？

- **工程体系**：有一套成熟的 Claude Code 工程规范（CLAUDE.md、Skills、MCP Server 组合），能复用到任何新项目；
- **作品可见**：GitHub 上有 10+ 个可演示项目、5+ 篇技术文章、形成可被外部验证的技术档案；
- **AI 研判力**：有自己的 AI 信息追踪系统，能独立判断什么工具/模型/框架值得投入，什么只是噪音；
- **交付能力**：能独立完成一个 AI-Native 项目的全流程闭环交付——需求→设计→实现→Eval→部署→复盘；
- **Agent 编排**：能设计和运行多 Agent 协作系统，实现单个 Agent 无法完成的任务。

## 信息 #1：Claude Code 工程化 —— Skills + Hooks + MCP 三件套成为区分普通用户和高级用户的明确分水岭

- **信息名称**：Claude Code 工程化体系成熟
- **信息来源**：
  - Anthropic 官方文档：Claude Code Hooks 系统、Skills 系统、MCP 协议
  - GitHub 社区：多个 Skills 模板仓库和 MCP Server 集合涌现
  - Claude Code CLI v2.0+ 推出的 Settings 分层管理（global/project/local）
- **这条信息说明了什么**：Claude Code 不再是一个"对话工具"，而是一个可以配置、可以扩展、可以编排的 Agent 操作系统。Skills 定义了 Agent 的专业能力模块，Hooks 定义了 Agent 的生命周期行为（PreToolUse、PostToolUse 等），MCP Server 打通了 Agent 和外部数据源的连接。三件套组合使用的人，和只会在终端打字聊天的人，差距是指数级的。
- **它为什么重要**：当大多数人还在争论"Claude 和 ChatGPT 哪个好用"的时候，真正的分层已经开始。这个分层不是模型能力的差距，而是**工程化能力的差距**。掌握三件套的人，一个人能完成一个团队的工作。
- **它可能适合什么人**：所有以 AI Agent 为核心工具的开发者、独立创业者、技术研究者和有 AI 工程化需求的学生。
- **它对我有什么启发**：我需要把自己的 CLAUDE.md 从一个简单的指令文件升级为一个完整的工程规范，包含 Skills、Hooks、MCP 配置的最佳实践。这本身就是一种壁垒。
- **它是否可能带来学习、比赛、项目或未来创业机会**：
  - 学习：Skills 本身就是可复用的知识模块，每学会一个新技能就沉淀为一个 Skill 文件
  - 比赛：AI Agent 比赛中，谁的 Agent 工程化程度高谁就有竞争优势
  - 创业：为特定行业（教育、金融、电商）定制 MCP Server + Skills 组合是一个可交付的服务产品
- **是否存在风险或不确定性**：Claude Code 迭代速度极快，今天的最佳实践明天可能就过时。需要在实践和跟进之间找到平衡。

## 信息 #2：AI Agent 编排框架爆发 —— 从单 Agent 到多 Agent 协作的范式跃迁

- **信息名称**：Multi-Agent 编排框架成为 2026 H1 GitHub 增长最快赛道
- **信息来源**：
  - GitHub Trending / Stars：CrewAI (47.8K★)、MetaGPT (59.6K★)、Microsoft Agent Framework 1.0（2026年4月 GA）
  - 阿里云天池 Agent 算法大赛（20万奖金 + 秋招直通）
  - AWS / 腾讯云黑客松全部转向 Agent 方向
- **这条信息说明了什么**：行业共识已经形成——单 Agent 的天花板很明显。幻觉、上下文窗口限制、单领域深度不足，这些都难以通过单一模型解决。解决方案是多 Agent 协作：每个 Agent 负责一个细分任务，互相校验、互相补充，最终协作完成复杂任务。
- **它为什么重要**：2026 年被很多团队称为"Agent 元年"。当单 Agent 能力趋于同质化（因为大家的底层模型差不多），差距就会出现在 Agent 编排层。会编排的人 vs 不会编排的人，就像会管理团队的人 vs 只会自己做的人。
- **它可能适合什么人**：有工程基础、需要处理复杂多步骤任务的开发者；创业团队中负责 AI 架构的人；有比赛和项目需求的学生。
- **它对我有什么启发**：我目前的 AI 使用方式主要是单 Agent 对话，还没有尝试过多 Agent 协作。这是 DEEP 营后续任务的重要方向，我应该提前储备框架知识（CrewAI、MetaGPT）并尝试设计一个简单的多 Agent 协作 demo。
- **它是否可能带来学习、比赛、项目或未来创业机会**：
  - 学习：多 Agent 系统的设计和调试是一套独立的能力，需要刻意练习
  - 比赛：Agent 算法大赛直接考察 Agent 编排能力
  - 创业：能设计垂直领域的 Multi-Agent 系统，本身就是一个高价值的服务能力
- **是否存在风险或不确定性**：多 Agent 系统的复杂度和调试难度远高于单 Agent，存在"过度工程化"的风险——用一个简单的脚本就能解决的问题不应该上多 Agent。

## 信息 #3：Web 自动化赛道成熟 —— Firecrawl、Playwright MCP、Browser-Act 构成的 AI 数据获取栈

- **信息名称**：AI 驱动的 Web 自动化工具栈已成熟，打通了"信息获取→数据处理→知识沉淀"的完整链路
- **信息来源**：
  - Firecrawl v2 API（搜索 + 抓取 + 提取 + Agent 多合一）
  - Playwright MCP Server（将浏览器自动化能力注入 Claude Code）
  - Browser-Act CLI（专用的 AI Agent 浏览器自动化工具，支持 stealth 指纹伪装和代理）
  - 多个 MCP Server 将 web scraping 能力标准化
- **这条信息说明了什么**：AI 不再只是生成文本，它现在能主动获取互联网上的实时信息。Firecrawl 提供了完整的搜索→爬取→结构化提取的能力，Playwright MCP 让 Claude Code 可以直接操控浏览器，Browser-Act 则提供了更专业的账号登录、多会话管理和反检测能力。这套工具体系打通了 AI 和数据源之间的最后一道墙。
- **它为什么重要**：信息获取能力是 AI 壁垒的核心组成部分。AI 可以做分析、做判断、做决策——但如果它依赖的训练数据是过时的，或者不能主动获取最新信息，那它的价值就大打折扣。掌握了 Web 自动化栈的人，相当于给自己的 AI Agent 装上了"实时感知器官"。
- **它可能适合什么人**：做数据分析、竞品监控、内容聚合、学术研究、自动化测试的任何人；需要从互联网获取结构化数据的 AI 应用开发者。
- **它对我有什么启发**：我目前已经在使用 Firecrawl MCP 和 Playwright MCP，但 Browser-Act 的反检测能力（stealth browser、代理网络）是我需要进一步探索的。把这个工具栈文档化、模板化，会是一个有价值的产出。
- **它是否可能带来学习、比赛、项目或未来创业机会**：
  - 学习：Puppeteer/Playwright + Firecrawl 是数据工程的核心技能
  - 项目：可以搭建一个 AI 驱动的自动信息聚合系统（如 AI 前沿资讯日报）
  - 创业：Web 数据采集和监控是一个成熟的 B2B 需求，AI 加持后效率提升 5-10 倍
- **是否存在风险或不确定性**：Web 自动化涉及法律合规问题（robots.txt、GDPR、网站 ToS），需要在使用时注意合规边界。

# 任务一复盘 — 戴昊杰

## 1. 完成这个任务后，我对 AI 的理解发生了什么变化？

完成这个任务的过程本身就是一次"用 AI 辅助深度思考"的实践。

最大的变化是：**我从"工具使用者"的视角切换到了"壁垒建设者"的视角**。之前使用 Claude Code，关注点更多在"这个任务能不能完成"、"代码能不能跑"。但写壁垒方向卡的时候，我开始系统性地回答一些更难的问题：我和别人的差距到底在哪里？我积累了什么东西是可以复用的？一个工具如果人人都会用，我凭什么比别人强？

这个切换让我意识到：**AI 时代最危险的舒适区不是"不会用 AI"，而是"会用 AI 但不会建壁垒"**。如果我只是把 Claude Code 当做一个更快的执行工具，我的使用水平和六个月后的新手不会有本质区别。但如果我把它当做一个可以持续沉淀知识资产的操作系统——CLAUDE.md 每优化一次都是一次资产积累，Skill 每沉淀一个都是一次能力固化——那我的壁垒就在时间复利中自然形成。

另一个重要变化是：我意识到**信息获取能力本身就是一种壁垒**。写"信息差观察"这个部分时，我重新审视了自己的信息来源——GitHub Trending、Twitter/X 上的 AI 讨论、Firecrawl 的搜索能力——发现这些信息源如果被系统性地利用，完全可以支撑一个有价值的 AI 信息追踪系统。目前我只是偶尔看看，没有被纳入日常 routine。

## 2. 我是否发现自己之前存在工具焦虑、信息焦虑或伪需求问题？

**工具焦虑**：存在，但不严重。我确实会关注新工具（Browser-Act、各种 MCP Server），但大部分时候能保持克制——先问"这个工具解决了我现在遇到的什么真实问题？"如果找不到答案，就先不投入。

**信息焦虑**：这是更真实的问题。DEEP 营的任务群里每天有大量信息，AI 领域的新闻更是层出不穷。我之前的信息处理方式是被动接收，而不是主动筛选。这个任务让我意识到：需要建立一个"信息筛网"——不是看得更多，而是看得更准。

**伪需求**：这个问题我需要认真反思。有时候我会"为了用 Claude Code 而想"——某个事情本来不复杂，但我偏要用 Agent 模式去做，结果花在配置和调试上的时间比手动完成的还多。这不是 AI 的问题，是我对"什么任务适合 AI 做"的判断还不成熟。

## 3. 我现在最应该优先学习哪 1-2 个 AI 工具？

**不增加新工具。深度优化现有工具栈。**

- **Claude Code（系统化升级）**：不是"学"一个新工具，而是把我已经使用的 Claude Code 从"对话工具"升级为"工程操作系统"。具体来说：完成 CLAUDE.md 的重构、Skills 体系搭建、MCP Server 的最优组合配置。这是我当前最大的杠杆点。

- **Firecrawl（从 MCP 调用升级为系统级使用）**：我目前通过 MCP Server 使用 Firecrawl 的搜索和抓取功能，但使用方式比较零散。我计划把它升级为一个系统化的信息获取工具：搭建 AI 前沿资讯自动追踪系统，从搜索→抓取→结构化提取→报告生成形成完整链路。这同时也是任务三（Agent 雏形搭建）的备选方向。

**为什么不是 CrewAI 或 MetaGPT？** 多 Agent 编排是重要方向，但在单 Agent 工程化还没到位的时候去学多 Agent 编排，是"走路还没稳就想跑"。先把 Claude Code 的工程化程度提上来，有了扎实的单 Agent 基础，再去探索多 Agent 协作才有意义。

## 4. 我目前最薄弱的壁垒是什么？

**作品的"可见性"和"可验证性"**。

诚实地说：我已经做了不少事情——几个 Web 项目、MCP Server 配置、自动化脚本、数据分析尝试。但这些东西大部分只存在于我自己的电脑上。外人无法验证我"到底会什么"。

在 AI 时代，能力不可见等于能力不存在。面试官不会问"你觉得自己会不会 Claude Code"，他们会说"给我看你的 GitHub、你的技术文章、你的开源项目"。

这是我接下来需要重点补的短板：**每做一个东西，都要有可公开访问、可被他人验证的产出**。

## 5. 下一步我准备如何通过任务二进入真实场景应用？

任务二（AI 真实场景应用实验）要求把 AI 放进一个真实场景中完成具体交付。我已经有了初步想法：

**方向**：搭建一个 **"AI 前沿信息自动化追踪日报"系统**。

**为什么选这个**：
- 它是一个真实需求（我自己就需要一个 AI 信息雷达）
- 它直接关联任务一的发现（信息差是核心壁垒）
- 它能展示 AI 全链路能力：Firecrawl 抓取 → Claude Code 分析/结构化 → Markdown/HTML 日报输出
- 它可交付、可展示、可复用（做完后我自己每天都在用）

**初步计划**：
1. 用 Firecrawl Search 定时搜索 AI 领域关键词（新模型、Agent 框架、MCP Server、比赛等）
2. 用 Claude Code 对抓取结果进行结构化分析和摘要
3. 生成一份 Markdown 格式的 AI 前沿日报
4. 可选：用 Claude Code 生成一个简单的 HTML 日报页面
5. 输出：完整的项目仓库 + 使用文档 + 样例日报

这个任务完成后，可以直接作为任务三（Agent 雏形搭建）的基础——在日报系统上加一个 Agent 自动化调度层，让它从"手动触发"变为"定时自动运行并推送"。这样三个任务形成一条完整的递进链，而不是三个孤立的交付。
